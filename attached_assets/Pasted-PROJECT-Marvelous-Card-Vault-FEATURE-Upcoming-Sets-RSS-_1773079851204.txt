PHASE 2 — RSS SYNC SERVICE
==================================================

Create a server service:

server/services/upcomingSetsSync.ts

SOURCE
Use ONLY the Cardboard Connection RSS feed:
https://www.cardboardconnection.com/feed

HARD RULES
- RSS feed only
- No HTML page scraping
- Do not surface source URLs anywhere in the user UI
- sourceUrl is internal storage only
- Do not modify DB schema

PACKAGE INSTALLS
If needed, install:
- fast-xml-parser
- node-cron

Use existing package manager conventions in the repo.

FUNCTIONAL REQUIREMENTS

Fetch and parse the RSS 2.0 XML feed.

Filter feed items using the following inclusion rules:
Match if title OR description contains ANY of the following, case-insensitive:
- marvel
- captain america cards
- avengers cards
- x-men cards
- spider-man cards
- topps marvel
- panini marvel
- upper deck marvel

Exclude if title OR description contains ANY of the following, case-insensitive:
- batman
- star wars
- pokemon
- baseball
- football
- basketball
- soccer
- hockey

For each matching item, extract:

1. setName
- from item title

2. sourceUrl
- from item link
- stored internally only
- NEVER rendered in UI

3. keyHighlights
- from item description
- strip all HTML tags
- decode HTML entities if needed
- normalize whitespace
- truncate to 500 chars max

4. releaseDateEstimated
- first try to parse a date from description using this regex:
  /(?:January|February|March|April|May|June|July|August|September|October|November|December)\s+\d{1,2},?\s+\d{4}/i
- if found, use that parsed date
- if not found, use pubDate + 60 days

5. dateConfidence
- "confirmed" if date found in description
- "estimated" if using pubDate fallback

6. manufacturer
Infer from title, case-insensitive:
- "Topps"
- "Panini"
- "Upper Deck"
- "Card Fun"
- "Wizards of the Coast"
Else use:
- "Unknown"

7. thumbnailUrl
Try in this order:
- media:content url
- enclosure url
- else null

8. default status fields
- status = "upcoming"
- isActive = true

DEDUPLICATION
Deduplicate by sourceUrl.
If sourceUrl already exists in DB, skip it.

Also, if the app already has a better existing uniqueness pattern, preserve it and add sourceUrl dedupe on top if possible without schema changes.

LOGGING
Log a single summary line like:
[RSS Sync] Added: X | Skipped (duplicate): Y | Skipped (not Marvel): Z | Errors: N

Return structured results from the service so admin endpoints can use them:
{
  added: number,
  skippedDuplicate: number,
  skippedNotMarvel: number,
  errors: number
}

SCHEDULING
Run this sync every 6 hours via node-cron.

Do not create duplicate cron registration if the server hot reloads or re-initializes.
Follow the app’s existing startup/service initialization pattern.

PHASE 3 — AUTO-EXPIRE RELEASED SETS
==================================================

Create a scheduled job that runs daily at 6:00 AM Central Time.

Behavior:
- Query upcoming_sets where:
  - releaseDateEstimated <= now
  - status != "released"
- Update matching records:
  - status = "released"
  - isActive = false

Log:
[Auto-Expire] Marked X sets as released

Also expose this logic as a callable service function so the admin endpoint can trigger it immediately.

IMPORTANT
Use the app’s existing date handling conventions.
Be careful with timezone handling.
The target schedule is 6:00 AM Central Time.
