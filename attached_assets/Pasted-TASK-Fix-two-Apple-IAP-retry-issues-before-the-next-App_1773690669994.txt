==================================================
FIX 1 — PREVENT LISTENER ACCUMULATION
==================================================

Problem:
Each retry may re-register store.when().productUpdated() and store.when().ready() listeners without cleaning up the previous ones.

Risk:
After retries, multiple listeners can stack and fire duplicate callbacks, causing race conditions and corrupted readiness behavior.

Requirements:
- Audit exactly how listeners are currently attached in appleIAP.ts
- Determine what the plugin actually returns from:
  - store.when().productUpdated(...)
  - store.when().ready(...)
- Do NOT assume unsubscribe() exists unless confirmed by the actual API behavior already in use
- Implement proper cleanup before re-registering listeners on retry
- If the API does not provide removable listener handles, refactor to a safe one-time listener strategy that avoids stacking
- Preserve existing readiness behavior:
  - loading
  - ready
  - failed

Goal:
A retry must not accumulate duplicate event handlers.

==================================================
FIX 2 — PREVENT DOUBLE-TAP RETRY
==================================================

Problem:
If the user taps the retry action twice quickly while readiness is still 'failed', preloadAppleIAP() may be triggered twice before the UI flips to 'loading'.

Requirements:
- In upgrade-modal.tsx, add a local retry lock such as isRetrying
- On retry tap:
  - if already retrying, do nothing
  - otherwise set retrying immediately and call preloadAppleIAP()
- Reset retry lock when readiness changes away from 'failed' or when loading begins
- Keep the retry UI lightweight and user-friendly

Goal:
User cannot trigger overlapping retry attempts by double tapping.
